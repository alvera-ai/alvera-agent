# Healthcare order-walk — per-phase detail

The 11 phases below mirror `integration-tests/tests/healthcare/_order.json`
at `@alvera-ai/platform-sdk@0.7.2`. Each phase has the same shape:

- **What** — the resource(s) provisioned in this phase
- **Skill** — which primitive skill the orchestrator hands off to
- **Elicit** — fields the conversation needs to surface
- **Verify** — the integration-tests spec the user can run as proof
- **Notes** — domain-specific gotchas

The phase tags (`bootstrap`, `data-sources`, …) match the spec slugs in
`_order.json`. Don't rename them.

## 1. bootstrap

- **What.** Tenant + datalake. Creates the workspace healthcare resources
  attach to. Writes the regulated + unregulated DB schemas, the
  regulated + unregulated S3 buckets, the bot user, and the
  Phoenix-side dataset migrations. Polls the datalake until
  `status === "ready"` (can take a couple of minutes on cold stacks).
- **Skill.** `/guided` — its datalake creation flow handles DB + S3
  credential elicitation and the migration-poll loop.
- **Elicit.** Tenant name, datalake name, regulated + unregulated DB
  hosts/users/passwords/schemas, S3 bucket names + region + credentials.
  Treat all secrets as env-var inputs, never inline in chat.
- **Verify.** `pnpm test:healthcare -- bootstrap`
- **Notes.** The tenant slug is auto-generated by the API
  (`vitest-e2e-healthcare-<runId>` in tests; user-friendly slug in
  prod). Do NOT re-create a tenant if one already exists for the user
  — list first, then either reuse or pick a different name.

## 2. data-sources

- **What.** A data source declares the *external system* the user is
  ingesting from (EHR, manual upload portal, etc.). The healthcare
  spec creates an "Athena Health EMR" source with URI
  `our-emr:prime-emr-source`. Multiple sources are common.
- **Skill.** `/guided` — `data-sources create` flow.
- **Elicit.** Source name, URI (a stable identifier the user picks —
  not the EMR's own URL), description, status (active/inactive),
  default flag.
- **Verify.** `pnpm test:healthcare -- data-sources`
- **Notes.** The URI is what later resources (DACs, workflows) reference
  to identify the source — collisions across the tenant are rejected.
  Suggest a `<system-id>:<source-id>` shape if the user is stuck.

## 3. custom-datasets

- **What.** A "generic table" — schema-on-read for data the user wants
  in the datalake but that isn't a FHIR resource. Healthcare commonly
  uses these for ops-side tables (appointment reviews, contact-us
  submissions, custom intake forms).
- **Skill.** `/custom-dataset-creation` — owns the compliance gate,
  column profiling, and schema proposal loop.
- **Elicit.** Sample file (CSV or NDJSON), table name, compliance class
  (dummy / shareable / regulated). All other fields are derived.
- **Verify.** `pnpm test:healthcare -- custom-datasets`
- **Notes.** If the user has multiple custom tables, run this phase
  multiple times (one table per invocation). Don't try to batch.

## 4. interoperability-contracts

- **What.** A Liquid template that maps source-system records →
  platform schema (FHIR resources or a custom dataset). Healthcare
  ships with two reference contracts: `Athena Appt Mapping`
  (appointments) and a "Contact Us" identity contract for the
  triage-AI flow.
- **Skill.** `/guided` — `interop-contracts create` flow. (For
  template *generation* from a sample file, `/DAC-upload`
  auto-generates them on demand — but the standalone phase here is
  about declaring named contracts for ops use.)
- **Elicit.** Contract name, target dataset (FHIR resource type or
  custom-table slug), Liquid template body, sample input row for
  preview.
- **Verify.** `pnpm test:healthcare -- interoperability-contracts`
- **Notes.** Date-format normalisation (MM/DD/YY → ISO 8601),
  gender code normalisation, and identifier extraction are the
  three most common transformations — call them out proactively
  if the user's source data shows any of them.

## 5. invite-team

- **What.** Invite collaborators to the tenant with appropriate roles
  (admin / writer / reader). Email-based invite flow.
- **Skill.** `/guided` — admin invite flow.
- **Elicit.** Email addresses + role per address.
- **Verify.** `pnpm test:healthcare -- invite-team`
- **Notes.** Optional phase. If the user is solo-dev'ing, skip with a
  one-liner: "Skipping invite-team — you can come back to this phase
  later."

## 6. tools

- **What.** Tools = effectful side-effect handlers. Healthcare commonly
  needs three: an SMS tool (SNS or Twilio), a "Manual Upload" tool
  (file drop), and at least one LLM tool (Ollama or OpenAI) for the
  AI-driven workflows in phase 11.
- **Skill.** `/guided` — `tools create` flow. Each tool type has its
  own creation form (SMS needs ARN + region; Manual Upload needs
  intent; LLM needs base URL + model).
- **Elicit.** Per tool: name, type, type-specific config. Loop through
  the three tools the workflow phases will need.
- **Verify.** `pnpm test:healthcare -- tools`
- **Notes.** SNS tool creation runs an STS GetCallerIdentity probe at
  create time — invalid creds reject with 422 here, before any
  workflow tries to send. Fail-fast is intentional; don't try to
  retry without fixed credentials.

## 7. create-dac

- **What.** A Data Activation Client — wires together a tool
  (data_exchange intent), a data source, and one or more interop
  contracts into an ingestion pipeline. The reference DAC is "CAHPS
  Manual Reconciliation" (manual-upload tool + EMR source +
  appointment + patient interop contracts).
- **Skill.** `/guided` — `data-activation-clients create` flow.
- **Elicit.** DAC name, tool slug (must have intent
  `data_exchange`), data source slug, list of interop-contract slugs.
- **Verify.** `pnpm test:healthcare -- create-dac`
- **Notes.** Phase 6 (tools) and 4 (interop-contracts) **must**
  succeed before this phase — DAC create asserts both exist by slug.
  If either is missing, the right move is to go back to that earlier
  phase, not to skip this one.

## 8. run-dac-single

- **What.** Ingest a single JSON row through the DAC end-to-end:
  presigned URL → upload → ingest-file → MDM merge → unregulated
  read returns a tokenised row. The smallest possible end-to-end
  proof of life.
- **Skill.** `/DAC-upload` — single-row mode (NDJSON with one record).
- **Elicit.** A single-row JSON file matching the DAC's source
  schema. If the user doesn't have one, point at
  `tests/healthcare/fixtures/` in platform-sdk@v0.7.2 for a sample.
- **Verify.** `pnpm test:healthcare -- run-dac-single`
- **Notes.** Uses Oban + a worker pool — ingest is async. The skill
  should poll `data_activation_logs` until status is settled
  (typically <30s).

## 9. run-dac-bulk

- **What.** Same pipeline as phase 8, but a multi-row CSV. Exercises
  the bulk-ingest path. Reference fixture is
  `tests/healthcare/fixtures/csv/alvera_reviews_cahps_appointments_batch1.csv`.
- **Skill.** `/DAC-upload` — bulk CSV mode.
- **Elicit.** A CSV file. If the user doesn't have one, point at the
  reference fixture.
- **Verify.** `pnpm test:healthcare -- run-dac-bulk`
- **Notes.** Bulk takes longer than single — 20-30s on the dev stack
  for ~3 rows. Don't time-box too tightly.

## 10. standard-workflow

- **What.** A template-driven workflow — the canonical example is
  "Review SMS after appointment": filter for fulfilled appointments,
  optional decision stage, SMS action via the SNS tool, idempotency
  via dedup window. Drafts → dry-runs → live-promotes.
- **Skill.** `/agentic-workflow-creation` — Review SMS template path.
- **Elicit.** Source URI (which DAC's output to consume), dedup
  window, SMS body Liquid template, target tool slug, optional
  filter expression.
- **Verify.** `pnpm test:healthcare -- standard-workflow`
- **Notes.** This phase's output is what the agent-driven workflow
  in phase 11 builds on — don't skip phase 10 expecting to "go
  straight to AI workflows".

## 11. agent-driven-workflow

- **What.** A workflow with an AI-agent enrichment stage between
  filter and action. Reference is "Contact Us triage": LLM
  classifies inbound messages, then routes to SMS / email / human
  via different action branches. Covers both the happy path
  (good Ollama URL) and the error path (bad URL → error.json
  carries `tool_execution_failed`).
- **Skill.** `/agentic-workflow-creation` — custom build path with
  AI-agent stage.
- **Elicit.** AI agent prompt + LLM tool slug, action mapping per
  classification, connected app slug for any magic-link URLs.
- **Verify.** `pnpm test:healthcare -- agent-driven-workflow`
- **Notes.** Two LLM tools are typical in this phase — one happy URL,
  one bad URL — to exercise both success and tool_execution_failed
  paths. The reference spec proves the bad-URL workflow lands
  `error.json` with structured detail; the conversational walk
  should call this out so the user verifies error handling, not
  just the happy path.

## After phase 11

The setup is complete. Next steps the orchestrator should mention:

- `/query-datasets` to verify rows landed in the datalake
- The full `pnpm test:healthcare` suite to re-run all 11 phases as
  one regression check
- `infra.yaml` (if the receipt was enabled) is the reproducible
  artefact of what was provisioned — keep it under version control

If the user wants to skip ahead to a different domain, point them
at the `accounts-receivable` or `payment-risk` skill (currently
stubs — they fall back to `/guided` for ad-hoc setup).
